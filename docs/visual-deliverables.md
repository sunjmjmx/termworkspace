# TermWorkspace · 每阶段可视物定义

> 在 Phase 启动前，SUN 确认每个阶段的「可视物」——看一眼截图就能判断"对/不对"的东西。
> 不在 Constitution 之外凭空增加工作量，截图就是 Worker 已有交付物的一部分。

## 可视物总表

| Phase | 可视物 | 形式 | 验收标准 |
|-------|--------|------|---------|
| **Phase2-1 流式响应支持** | TUI 运行截图：发送消息后，回复逐 chunk 出现的过程 | PNG 截图（可串联成 GIF） | 能看到助理回复的文本逐 chunk 在对话区追加，而非一次性显示全部 |
| **Phase2-2 代码健壮性提升** | 异常状态截图 × 3：网络断开 / API Key 无效 / 配置文件缺失 | 3 张 PNG 截图 | 每个异常有对应的错误提示信息，UI 不崩溃，有明确的"如何修复"指引 |
| **Phase2-3 Workspace 模板市场** | 导出/导入流程截图：选择模板 → 确认 → 完成提示 | 2-3 张 PNG 截图（串联流程） | 导出文件路径可见，导入后 workspace 配置正确加载 |
| **Phase2-4 会话持久化与恢复** | 前后对比截图 × 2：关闭程序前的对话界面 → 重启后自动恢复的界面 | 2 张 PNG 截图 | 重启后对话内容完整恢复，模型选择、历史记录与关闭前一致 |
| **Phase2-5 打包与分发** | CLI 输出截图 + 安装过程终端截图 | PNG 截图 × 2 | `termworkspace --help` 输出完整；`pip install` 过程无报错 |

## 截图规范

1. **文件命名**：`{phase-name}-{场景}.png`，如 `streaming-before.png`、`streaming-after.png`
2. **保存位置**：`docs/screenshots/{phase-name}/` 目录下
3. **截取方法**：Worker 运行 TUI 后，用 macOS 系统截图（`screencapture`）或 Python 截取终端窗口
4. **对比优先**：有"改前 vs 改后"或"正常 vs 异常"对比的场景，两张并列优于单张

## 审核流程

```
Worker 开发完成
  → block review-required
  → SUN 看截图（3-10 秒判断）
    → ✅ 截图符合验收标准 → 继续审查代码和文档
    → ❌ 截图不符合 → 直接在 comment 里说"缺哪个场景的截图/截图看不清"
```

这样 SUN 的审核时间从「读完代码才判断」降为「看一眼截图就能反馈」。
