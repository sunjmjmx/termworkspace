## 四、可沉淀解决方案

### 4.1 技术方案

#### forward_to() 方法（window.py）

```python
def forward_to(self, messages: list[dict], source_name: str = "") -> None:
    """Inject forwarded messages from another panel into this one."""
    if not messages:
        return
    for msg in messages:
        entry = {"role": msg.get("role", "user"), "content": msg.get("content", "")}
        if source_name:
            entry["forwarded_from"] = source_name
        self.messages.append(entry)
    # ... UI update with TextArea ...
```

#### 兄弟面板发现模式（app.py）

```python
def _wire_panel_forward_targets(self, panel: AIWindowPanel) -> None:
    """Walk widget tree up to WorkspaceView, find sibling panels."""
    parent_view = None
    try:
        current = panel.parent
        while current is not None:
            if isinstance(current, WorkspaceView):
                parent_view = current
                break
            current = current.parent
    except Exception:
        pass
    if parent_view is None:
        panel.set_forward_panels([])
        return
    siblings = parent_view.panels
    targets = []
    for sibling in siblings:
        if sibling is panel:
            continue
        targets.append({"id": sibling.id, "label": f"Panel {sibling._panel_index}"})
    panel.set_forward_panels(targets)
```

## 五、向下游传递的信息

### 5.1 已知问题（待后续 Phase 处理）

- 转发目标下拉框目前只是 `Panel {index}` 格式，未来可显示模型名或自定义名称
- `forward_to()` 中 `history.text` 拼接在高频转发时可能有性能问题（TextArea 增量追加）

### 5.2 架构决策

- **Widget tree 遍历而非注册表**：`_wire_panel_forward_targets` 通过 `panel.parent` 向上查找 `WorkspaceView`，不引入新的管理器或注册中心
- **转发保留完整消息列表**：转发整个 `source_panel.messages`，而非仅最后一条，支持批量上下文传递
- **转发源标记**：每条被转发消息加 `forwarded_from` key，不改变原有 `role`/`content` 结构

### 5.3 后续建议

- P4-2（Agent 模式）直接复用本任务的 `forward_to()` + `on_ai_window_panel_forward_requested` 接口
- 考虑在 forward dropdown 中排除 source panel 自身的逻辑已在 `_wire_panel_forward_targets` 中实现（`if sibling is panel: continue`）
- 若未来需要跨 tab 转发，当前的 widget tree 遍历策略需要扩展（当前只遍历到所属 WorkspaceView 为止）

## 六、小结

P4-1 的架构设计简洁：window.py 定义数据接口 + 消息类，app.py 负责发现兄弟面板路由 + 处理转发请求，两者通过 Textual 消息机制解耦。最大教训是 Worker 的多轮提交容易产生 revert 累积误差——需要强制的 pre-handoff 验证流程来防止 `handoff 说做了但代码没有` 的问题。
